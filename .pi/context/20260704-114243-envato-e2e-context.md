### Pipeline Path Audit
# Pipeline Path Audit
Generated: 2026-07-04

## Runtime path today

- `deck.json` → `shared/pptx/schema.py::load_deck()` → JSON Schema validation + semantic validation.
- `shared/pptx/build.py::build_deck()` orchestrates cloning from `templates/template.pptx`, applies chrome slots, optionally runs `expand_layout()`, then dispatches to `render_block()`.
- `shared/pptx/layouts.py` currently supports only 3 semantic layouts: `gantt`, `comparison_panel`, `kpi_strip`.
- `shared/pptx/blocks.py` currently supports only 10 block builders: `heading`, `body`, `bullets`, `caption`, `table`, `card`, `darkcard`, `steps`, `kpi`, `gantt`.
- `tools/pptx_validate/cli.py::validate()` checks brand compliance, chrome rules, fonts, colors, logo/footer positions, bounds, and round-trip integrity.

## Where palette/library is used today

**Key finding:** `templates/media/reference/library` is **not used by the runtime generator at all**.

- The library/palette is currently an **offline reference corpus** built by `scripts/media_library.py` and `tools/envato_assets/*`.
- The generator never queries `reference/library`.
- The validator never validates against library/widget semantics.
- The only runtime “palette” is `templates/design_tokens.yaml`, which is a brand token store, not a widget catalog.

## Gaps vs desired end-to-end palette-driven flow

1. **No widget selection API** — nothing can ask “which timeline/gantt widget variant fits this content?”.
2. **No library → layout mapping** — layouts emit block dicts directly, never library references.
3. **No library-aware validation** — no check that rendered output matches widget archetypes from the palette.
4. **Taxonomy mismatch** — library has 20 categories; runtime block kinds are only 10, with limited overlap.
5. **No runtime bridge for Envato metadata** — slot_count/orientation/text_capacity/etc. are generated offline but ignored at render time.

## Relevant files

- `shared/pptx/build.py` — main insertion point for any future palette-driven widget lookup.
- `shared/pptx/layouts.py` — current semantic layout registry.
- `shared/pptx/blocks.py` — current rendering taxonomy.
- `scripts/media_library.py` and `tools/envato_assets/*` — library generation, classification, and QA.

## Core conclusion

The current library/palette is a **dead-end reference corpus** relative to runtime generation. Any plan promising true end-to-end palette participation must include architecture work, not just tests.

### Master Isolation Audit
# Master Isolation Audit
Generated: 2026-07-04

## Intended separation model

The repo has a clean 4-layer separation:
1. `templates/template.pptx` — locked master template.
2. `templates/design_tokens.yaml` — machine-readable brand source of truth.
3. `clients/_sample/` — committed generic examples only.
4. `clients/<engagement>/` — local per-customer working decks, gitignored.
5. `templates/media/` / library / Envato pipeline — separate asset domain.

## Evidence of safeguards

- `.gitignore` blocks `clients/*` except `_sample/` and `clients/README.md`.
- `*.pptx` is ignored except `templates/template.pptx`.
- Generator CLI requires explicit `--schema`, `--template`, `--tokens`, `--out` paths.
- Tests hardcode `_sample`, not customer decks.
- No runtime code scans `clients/*` directories.
- Envato/media-library pipeline is confined to `templates/media/`.

## Possible contamination risks

1. Human error: someone edits `_sample` with customer content.
2. Local-only discovery risk: client directories exist on disk and an agent could read them if prompted.
3. Future risk: `deck_dir` is accepted by layout builders and could become a path traversal source if later used unsafely.
4. No automated check prevents customer content from creeping into `_sample`.
