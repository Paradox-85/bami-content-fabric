## 2. Where `technical-description.md` should mention neighboring sessions as accepted architecture evolution

### Current state
The word "neighboring" / "neighbouring" appears **nowhere** in `technical-description.md` (confirmed by grep). The concept of **phase-2 evolution** is mentioned only in §12.5 (Phase 2 future direction), which talks about re-authoring templates with placeholders — a single-template evolution, not a multi-session architecture.

### What's missing
The system currently assumes **one deck process owns the output file**. In practice, BAMi engagements may produce multiple related presentation streams that share template chrome but differ in content depth, audience, and purpose. For example:
- A "core" deck → executive summary
- A "detailed" deck → technical deep-dive for SME audience  
- A "workshop" deck → interactive, annotation-heavy

These are **neighboring sessions** — different deck.json files, same template, possibly sharing media assets, Mermaid caches, and validation gates. The architecture does not forbid this, but it never documents it as a supported pattern.

### Where to add it

**Insert a new short subsection** (maybe §12.6 or expand §12.5) in `docs/architecture/technical-description.md`:

> **12.6 Neighboring sessions (multi-deck engagements)**  
> The framework supports multiple deck outputs from the same template without architectural changes. Authors create separate `deck.json` files in `clients/<engagement>/` (or subdirectories like `clients/<engagement>/executive/`, `clients/<engagement>/technical/`), each independently passing through the generator + validator pipeline.  
>  
> Shared resources that work across neighboring sessions:  
> - `templates/template.pptx` — single source of chrome for all decks  
> - `templates/media/` — common image and reference pool  
> - `.pi/mermaid-cache/` — cache reuse across sessions for the same diagram definitions  
>  
> Each session produces its own `.pptx`; there is no merge step and no cross-session coupling. The framework treats every deck generation as an isolated transaction.

### Why in `technical-description.md` and not `README.md`
Because neighboring sessions are an **architectural consideration** (scalability, shared resources, isolation guarantees), not part of the quickstart or authoring model.

## 3. Where to explain media-library as a design-reference / graphic-pattern injection subsystem

### Current state
`technical-description.md` §5.5 describes `templates/media/` accurately as:
1. shared runtime image pool
2. flat visual benchmarks
3. categorized reference library

And mentions the pipeline output: category folders, QA artifacts, etc. But it frames the library as a **catalog artifact** — "here's what exists" — rather than as a **design-reference injection system** that feeds graphic patterns into the slide authoring process.

`README.md` describes the pipeline but similarly stops at cataloging.

### What's missing
The media library is not just a passive archive. Its real role in the architecture is:

1. **Design-reference input** — a human or agent authoring a `deck.json` browses `templates/media/reference/library/<category>/` to see visual examples of how `gantt`, `comparison_panel`, `kpi_strip`, `timeline`, `flow`, etc. are composed. The per-category READMEs describe *structure*, *reusable elements*, and *what to ignore* — turning each library entry into an authoring reference card.

2. **Graphic-pattern injection via envato_assets** — `tools/envato_assets/` ingests external stock graphics (Envato bundles), processes them through crop/classify, feeds into the media_library pipeline, and the final categorized PNGs end up in the library as **structured visual examples** that inform or inspire **body-zone block composition** in future decks.

3. **Benchmark source for validator / layout regression** — the flat `reference-*.png` benchmarks are used (or will be used) for visual regression: render a generated slide to PNG, compare against the benchmark to detect layout drift.

### Where to add it

**In `technical-description.md`, after §5.5** (or as a new §5.6 "Design-reference pipeline"):

> ### 5.6 Design-reference and graphic-pattern injection
> The media library at `templates/media/reference/library/` serves a role beyond passive asset storage. It is an **authoring surface** — a structured, categorized, visually searchable set of composition patterns that authors (human or agent) consult when choosing layouts and block configurations for a new deck.
> 
> The injection flow has three stages:
> 1. **Ingest** — `tools/envato_assets/` downloads, extracts, and pre-classifies external stock graphics from Envato Elements. Output lands in `templates/media/_envato_ingest/`.
> 2. **Catalog** — `scripts/media_library.py` inventory → classify → convert → finalize produces categorized PNGs under `library/<category>/` with per-category READMEs that document each file's structure, reusability, and ignore-notes.
> 3. **Consume** — during deck authoring, `library/<category>` entries serve as visual pattern references. The `SKILL.md` and per-category READMEs guide which patterns are safe for BAMi-like composition and which must be discarded (non-BAMi palette, decorative elements, etc.).
> 
> This pipeline decouples graphic-research (broadly sourcing inspiration) from slide-authoring (narrowly composing within the BAMi design system).

## 4. Whether current docs understate accepted review caveats (e.g. Mermaid cache concurrency note)

### Current state
`technical-description.md` §11.2 lists:

> - the Mermaid cache is adequate for normal single-process deck generation, but the current deterministic temp-output naming still deserves a future hardening pass for same-key concurrent renders.

This is **accurate** but **understated**. Here's why:

The current `mermaid_render.py` uses `os.replace(tmp_out_name, cache_path)` which *is* atomic on POSIX and Windows (same-filesystem rename is atomic), so **two concurrent cache-miss renders for the same diagram key cannot corrupt the cache** — the last writer's output wins. The real risk is:

1. **Two concurrent renders for the *same* key** waste CPU (both run mmdc), but are safe — the cache ends up with a valid PNG.
2. **Two concurrent renders for *different* keys but using the same tempfile names** — impossible, each gets a unique `tempfile.NamedTemporaryFile`.
3. The actual caveat is that `mmdc` itself has browser overhead (Playwright headless); running four concurrent deck builds on the same machine could exhaust memory on resource-constrained hosts.

The phrase "deterministic temp-output naming still deserves a future hardening pass for same-key concurrent renders" slightly overstates the risk — the atomic `os.replace` already prevents corruption. The real vulnerability is **performance under concurrent load**, not correctness.

### Recommendation
Reframe the caveat in `technical-description.md` §11.2 from:

> "the current deterministic temp-output naming still deserves a future hardening pass for same-key concurrent renders"

To:

> "the Mermaid render path uses atomic `os.replace` for cache publishing, so concurrent cache-miss renders for the same key cannot produce a corrupted file. However, each `mmdc` subprocess launches a headless browser; running multiple concurrent deck builds may exhaust memory on small hosts. A future hardening pass could add file-level locking to avoid redundant mmdc invocations for the same key."

### Whether README needs the same
No. The README's Mermaid limitations bullet already says "Mermaid diagrams are static images and are not automatically BAMi-themed" — that's sufficient for the README audience.
