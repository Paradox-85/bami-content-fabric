## Research Findings

### Template Architecture
Source artifact: `.pi/research/20260702-151126-template-architecture.md`

#### How generation works today
- Generator inputs:
  1. `templates/template.pptx` — locked reference deck with 8 reference slides.
  2. `templates/design_tokens.yaml` — source of truth for brand/system tokens and template slot mappings.
  3. `clients/<engagement>/deck.json` — content model with `{title, slides[]}`.
- Pipeline in `shared/pptx/build.py`:
  1. clone reference slide via `shared/pptx/clone.py`
  2. clear body zone on content slides only
  3. replace chrome slots via `shared/pptx/chrome.py`
  4. render free-positioned body blocks via `shared/pptx/blocks.py`
  5. prune original reference slides
  6. save deck
- Validator in `tools/pptx_validate` enforces Montserrat, brand colors, branded background/logo/footer, content title bar/title, bounds, and structural consistency.

#### Current philosophy / constraints
- Core principle repeated across skill/docs: **"Composition may vary; the system does not."**
- Exactly three slide templates today:
  - `cover`
  - `content`
  - `closing`
- `content` is the only hybrid template (slot title + free body zone).
- `cover` and `closing` are slot-only.
- `python-pptx` limitations drove the current architecture: no masters/layouts import, no layout creation, no font embedding.
- Chrome is inherited from clone, never recreated programmatically.

#### Where variability is intentionally limited today
- Only 3 templates.
- Only 9 block kinds: `heading`, `body`, `bullets`, `caption`, `table`, `card`, `darkcard`, `steps`, `kpi`.
- No nested composition system.
- No explicit slide-level layout taxonomy.
- No image/media blocks.
- No template inheritance or body-layout inheritance.

#### Risks/tensions if variability grows
- Clone-source explosion if every new layout becomes a new reference slide.
- Current body-zone clearing heuristic (`1.0"–10.5"`) will break if future variants have chrome/structural elements inside the body zone.
- Slot replacement is shape-name fragile.
- Validator currently has limited template awareness.
- Font embedding remains unresolved.

### Code Extension Points
Source artifact: `.pi/research/20260702-151126-code-extension-points.md`

#### Relevant module map
- `shared/pptx/tokens.py` — typed access to `design_tokens.yaml`
- `shared/pptx/schema.py` — inline schema + semantic validation
- `shared/pptx/blocks.py` — atomic body block constructors + `BUILDERS`
- `shared/pptx/chrome.py` — slot text replacement
- `shared/pptx/clone.py` — slide deep-clone + image rel remap
- `shared/pptx/build.py` — main orchestration and branching
- `shared/pptx/style.py` — typography/colors/shape styling helpers
- `tools/pptx_validate/cli.py` — brand/structure validator
- `schemas/content-schema.json` — mirrored persisted schema

#### Existing assumptions baked into code
- Hardcoded template enum: `cover`, `content`, `closing`
- Only `content` slides clear body zone and render blocks
- Cover/closing cannot have blocks
- Body zone is a fixed vertical band
- First slide must be cover, last must be closing

#### Main extension seams
1. **New atomic blocks**
   - Add constructor in `shared/pptx/blocks.py`
   - Register in `BUILDERS`
   - Extend both `shared/pptx/schema.py` and `schemas/content-schema.json`
   - Optionally extend validator if block introduces new visual semantics
2. **New slide archetypes / template variants**
   - Add reference slide(s) in `templates/template.pptx`
   - Add template entry in `templates/design_tokens.yaml`
   - Extend template enum + semantics in schema
   - Refactor `build.py` control flow beyond `if tname == "content"`
   - Extend validator for new chrome/body expectations
3. **Layout-selection logic**
   - Add slide-level properties such as `layout`, `style`, `intent`, etc.
   - Teach generator to interpret them before block rendering
4. **Image/media support**
   - Currently missing entirely
   - Likely needs new helper module such as `shared/pptx/media.py`
5. **Validator refactoring**
   - Current cover/content detection relies on logo-position heuristics
   - This will not scale to many new archetypes

#### Important technical risks
- Two schema sources must stay in sync (`schema.py` and `content-schema.json`).
- `ref_index` in `design_tokens.yaml` is fragile if template slide ordering changes.
- No auto-layout engine exists today; all blocks are explicitly positioned.
- New images require relationship management, not just shape creation.

### Slide Purpose Taxonomy from Historical PPTX Corpus
Source artifact: `.pi/research/20260702-151126-slide-purpose-taxonomy.md`

#### Corpus reviewed
- `Automation Activities.pptx`
- `BAMI - BIM & DIGITAL Services.pptx`
- `BAMI Company Profile Technip.pptx`
- `BAMI Digital Construction - Kanadevia.pptx`
- `BAMI Geotracking.pptx`
- `BAMI Meeting Update ROSETTI.pptx`
- `Deep Analysis ENI-General.pptx`
- `environemnt_architecture.pptx`
- plus `Presentation Template.pptx` for contrast
- Total scanned: 142 slides

#### Main slide-purpose groups identified
1. **P1 — Title / Cover**
   - Always present; already supported.
2. **P2 — Table of Contents / Agenda**
   - Numbered sections; common enough to become an archetype.
3. **P3 — Company Identity / About**
   - Variants: intro/tagline, history/timeline, vision/mission.
4. **P4 — Service / Capability Summary Grid**
   - 2×2 or 4-card grid of service families/capabilities.
5. **P5 — Service / Solution Detail**
   - Dominant pattern across corpus.
   - Structure: summary left + goals/features right + experience bar bottom.
6. **P6 — Architecture / Data Flow Diagram**
   - Highly custom, but recurring as a slide intent.
7. **P7 — Data / KPI / Dashboard**
   - Status metrics, data tables, visual scorecards.
8. **P8 — Project Status / Update / Roadmap / Next Steps**
   - Progress, remaining activities, roadmap, actions.
9. **P9 — Contact / Closing**
   - Reusable; already supported.

#### Secondary patterns with generator value
- **Section Divider**
- **Side-by-Side Comparison**
- **Process / Workflow Steps**

#### Most important takeaway
- The dominant reusable archetype is **P5 Service Detail**:
  - title band / service name
  - left summary narrative
  - right goals/features block(s)
  - bottom “experience with” / technology strip
- This is the highest-value gap between the current generator and the real corpus.

#### Deck genres observed
- **Service Catalog** — company/profile/service portfolio decks
- **Solution Pitch** — client/solution-specific proposals
- **Client Meeting / Update** — status/roadmap/feedback/next steps decks

### Layout Patterns and Atomic Content Primitives
Source artifact: `.pi/research/20260702-151126-layout-patterns.md`

#### Candidate reusable layout patterns
1. **Cover / Title**
2. **Section Divider**
3. **Left-Right Summary + Features** (most frequent content pattern)
4. **Four-Column Numbered Pillars**
5. **Text + Image Balanced**
6. **Architecture / Flow Diagram**
7. **Gantt-like / Timeline Band**
8. **Table / Comparison Grid**
9. **Closing / Contact**
10. **Bullet List / Feature Grid**

#### Candidate atomic content primitives
- Numbered step circle / numbered badge
- Feature card
- Callout strip / accent banner
- Milestone card / step card
- Legend block
- Quote block
- Dark emphasis card
- KPI / metric block
- Process arrow connector
- Horizontal separator line
- Icon group
- Row labels for diagrams/timelines

#### Pattern ↔ intent mapping highlights
- **Service/Product Overview** → left-right summary+features
- **Capability Showcase** → four-column pillars
- **Case Study / Demo** → text+image
- **Architecture Explanation** → connected-box flow diagram
- **Timeline / Roadmap** → gantt/timeline band
- **Metrics / Results** → KPI dashboard / scorecard
- **Process / Methodology** → steps / numbered flow
- **Section Transition** → section divider

#### Biggest gaps versus current generator
- No `image` block
- No `two_column` layout/container
- No `separator` / accent line block
- No `numbered_badge` primitive
- No `icon` primitive/menu
- No `timeline` / gantt primitive
- No standalone legend block
- No first-class quote block
- No architecture connector primitives

#### Suggested future LLM-facing menu concept
- Introduce higher-level slide `intent`, e.g.:
  - `overview`
  - `showcase`
  - `case_study`
  - `architecture`
  - `comparison`
  - `timeline`
  - `metrics`
  - `process`
  - `section`
  - `closing`
  - `company_intro`
- Map `intent` → template → default block composition → expected slots/content fields.
- LLM should classify content first, then select an archetype/layout, then fill slots/blocks.
