# Renderer Operating Model

> **Purpose:** Document the production operating model for the dual-renderer architecture.
> This is the authoritative policy for *which* renderer is primary, which is secondary,
> and how the two interact.
>
> **Last updated:** 2026-07-09

## Renderer Hierarchy

| Priority | Renderer | Role | Delivery target | Status |
|----------|----------|------|----------------|--------|
| **Primary (P0)** | Branch B — python-pptx | Production delivery renderer | `.pptx` (client-facing) | **Stable** — ships to clients today |
| **Secondary (P1)** | Branch A — Slidev / Vue | Fallback / preview / secondary renderer | Web SPA, PDF export | **Operational** — 8 Vue components, E2E pipeline |
| **Tertiary (P2)** | Mermaid → PNG (mmdc) | Temporary rendering bridge | PNG embedded in PPTX | **Bridge** — not target-state for key client widgets |

## Policy Statements

### 1. Branch B is the primary delivery renderer

All customer-facing deliverables are `.pptx` files generated by the python-pptx (Branch B)
pipeline. This is the renderer that must be production-stable, regression-free, and
continuously maintained. No Branch A work shall destabilize Branch B.

### 2. Branch A is a fallback / preview / secondary renderer

Branch A (Slidev / Vue) is used for:
- **Web SPA previews** — interactive browser-based review of deck content.
- **PDF export** — standalone PDF version of a deck for non-PowerPoint stakeholders.
- **Development preview** — agent/local testing before running the Branch B pipeline.
- **Quality upgrade path** — Vue components replace Mermaid→PNG rasterization for
  diagram categories (funnel, decision-tree, swimlane, mind-map, checklist) with
  resolution-independent SVG/HTML rendering.

Branch A is **not** a delivery renderer for client PPTX. It must not be treated as
"the replacement" for Branch B.

### 3. Mermaid is a temporary bridge, not target-state

Mermaid diagrams rendered via `mmdc` (Mermaid CLI → PNG) in Branch B are a **temporary
bridge** for categories that lack native PPTX blocks. The target state is:

1. **Native PPTX block** — the canonical Branch B solution.
2. **Vue component** — the canonical Branch A solution.

Mermaid→PNG should be retired for any category once a native PPTX block exists.
Prioritize categories where Vue already proves the structure (`funnel-diagram`,
`decision-tree-flowchart`, `swimlane-diagram`, `mind-map-radial`, `checklist-status`).

### 4. The no-implicit-fallback rule

A renderer must not silently approximate a category it does not support. See
`docs/architecture/renderer-ownership-matrix.md` for the detailed matrix.

### 5. Roadmap priority order

1. **PPTX-native coverage** — new blocks in `shared/pptx/blocks.py` and native PPTX
   rendering (no Mermaid dependency).
2. **Fallback branches** — Vue components and Mermaid bridges where native PPTX is
   not yet implemented.
3. **Web/PDF polish** — Slidev styling, export quality, and interactive features.
