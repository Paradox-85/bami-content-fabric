## 1. End-to-End Pipeline

### 1.1 Data flow (three inputs, one output)

```
templates/template.pptx   (locked 8-slide corporate deck)
templates/design_tokens.yaml  (single source of truth for geometry + palette)
clients/<engagement>/deck.json  (content model: slides, fields, blocks)

  ↓  build_deck(deck_path, out_path, template_path, tokens_path)

branded.pptx   (pruned: only rendered slides survive)
```

### 1.2 Pipeline stages (shared/pptx/build.py → `build_deck()`)

The orchestrator function `build_deck()` (line 51–109) runs in seven ordered
stages:

| Stage | Function / Location | What it does |
|-------|-------------------|--------------|
| **1. Parse inputs** | `schema.load_deck()` (schema.py:52) | JSON-loads deck.json, runs JSON Schema structural validation, runs semantic validation (first/last slide, cover/closing rules, section_divider guard) |
| **2. Load tokens** | `tokens.load_tokens()` (tokens.py:64) | YAML-loads design_tokens.yaml → `Tokens` typed accessor |
| **3. Open template** | `Presentation(str(template_path))` | python-pptx opens the locked .pptx |
| **4. Resolve references** | loops over `tokens.templates` → `ref_index` | Caches each template's source slide by index (cover=0, content=1, closing=7) before any clone |
| **5. Per-slide: clone → clear → fill → compose → hint** | for each `slide_spec` in `deck["slides"]` | See §1.3 below |
| **6. Prune reference slides** | `delete_slide_at(prs, 0)` × n_orig | Removes all 8 original reference slides from the front of the deck |
| **7. Save** | `prs.save(str(out_path))` | Writes the final .pptx |

### 1.3 Per-slide rendering loop (stage 5 detail)

For each slide spec in the deck:

```python
# build.py lines 80–103
new_slide, _ = clone_slide(prs, refs[tname])
if _capability(tokens, tname, "body_clears"):
    _clear_body_zone(new_slide, top, bot)  # removes shapes in y=[1.2, 10.5]
apply_slots(new_slide, tmpl.get("slots", {}), slide_spec.get("fields", {}))
_write_archetype_hint(new_slide, tname, ...)  # writes to slide notes
if _capability(tokens, tname, "has_blocks"):
    for block in slide_spec.get("blocks", []):
        render_block(new_slide, tokens, block, tname)
# layout expansion (stubbed for Phase C) — passes through
```

### 1.4 Validation pipeline (tools/pptx_validate/cli.py → `validate()`)

Re-opens the generated .pptx and checks every slide:

1. **Archetype hint**: reads `BAMI::template=<tname>` from slide notes (written
   by `_write_archetype_hint()` in build.py). This is the *authoritative* way
   to determine whether a slide is content vs cover/closing. Falls back to
   logo-position heuristics only when hints are absent.
2. **Background**: full-bleed picture (L=0, T=0, W=20, H=11.25).
3. **Logo**: at either content-position or cover/closing-position EMU coords.
4. **Fonts**: every text run must have `Montserrat` (case-insensitive).
5. **Colors**: every run and fill color must be in `brand_hexes` (the 13-color palette).
6. **Title bar** (content slides only): black `#0A0A0A` rectangle at (0, 0, 8.6, 0.95).
7. **Title text** (content slides only): Montserrat 24 bold `#FFFFFF` at y≈0.
8. **Footer**: "DELIVERING VALUE" + "Proprietary & Confidential".
9. **Canvas bounds**: every shape must be entirely within 20.0×11.25".
10. **Round-trip**: saves and re-opens the .pptx to catch corruption.
11. **Structure**: first slide must be cover-like, last must be closing-like.

Exit code 0 = pass; exit code 1 = violations printed to stderr.

## 2. Dependencies, Libraries & python-pptx Limitations

### 2.1 Hard dependencies (from pyproject.toml)

| Package | Version | Role |
|---------|---------|------|
| `python-pptx` | 1.0.2 | All slide manipulation, shape reading, saving |
| `pyyaml` | ≥6.0 | Parse `design_tokens.yaml` |
| `jsonschema` | ≥4.20 | Structural validation of `deck.json` against `content-schema.json` |
| `click` | ≥8.1 | CLI argument parsing (both pptx_gen and pptx_validate) |

Dev: `pytest>=8.0`, `ruff>=0.4`.

### 2.2 python-pptx hard limits (from ADR-0001)

These are the four documented python-pptx limits that the slide-clone strategy
avoids:

| Limitation | python-pptx issue | Mitigation in this framework |
|-----------|-------------------|------------------------------|
| Cannot create masters/layouts | #413, #656 | Slide-clone (deep-copy XML) instead of `add_slide(layout)` |
| Cannot embed fonts | #355 | Template references Montserrat by name; one-time PowerPoint UI embed recommended |
| Cannot import layouts across files | #1028 | Slide-clone copies within the same file |
| No built-in slide clone | — | `shared/pptx/clone.py` implements bit-faithful deep-clone with image-relationship remapping |

### 2.3 Known workarounds

- **Connector lines** (blocks.py `add_flow`): built via raw XML (`lxml` +
  `p:cxSp + a:ln` + `a:xfrm`) because python-pptx has no connector shape API.
  Only straight connectors are supported.
- **Image aspect ratios**: handled manually in `add_image()` with `PIL` (Pillow)
  for `fit=contain/cover/fill` modes. Pillow is *not* listed in pyproject.toml
  but is imported in blocks.py — this is a latent missing-dependency risk.
- **Slide notes access**: `slide.notes_slide` requires a notes master in the
  template. The `_write_archetype_hint()`/`_read_archetype_hint()` functions
  are wrapped in try/except for resilience.

## 3. Deck Schema, Template Dispatch, Tokens, Chrome Replacement & Block Rendering

### 3.1 Deck JSON Schema (`schemas/content-schema.json`)

- **Top-level**: `{title: string, schema_version?: integer (default 2), slides: array}`
- **Slide**: `{template: enum[cover, content, closing, section_divider], fields?: object, layout?: string, variant?: object, content?: object, blocks?: array}`
- **Block**: `{kind: enum[20 kinds], x, y, w (required), plus per-kind optional fields}`
- Loaded at import time into `schema.SCHEMA` dict (schema.py:33–38).
- Test `test_schema_sync.py` verifies the JSON Schema's block kind enum matches
  the `BUILDERS` dict in blocks.py — they must stay in sync.

### 3.2 Semantic validation (schema.py `_validate_semantics()`, line 58)

| Rule | Enforcement |
|------|-------------|
| At least one slide | Hard error |
| First slide must be `cover` | Hard error |
| Last slide must be `closing` | Hard error |
| `cover` may not appear in slides[1:-1] | Hard error |
| `closing` may not appear in slides[0:-1] | Hard error |
| `section_divider` is **always rejected** | Hard error — branded chrome not yet authored in template |
| Content slides must have `fields.title` | Hard error |
| Only content/section_divider may have `blocks` | Hard error |

### 3.3 Schema migration (schema.py `_migrate_deck()`, line 112)

- Current schema version: **2**.
- Version 0 (legacy) → 2: stamps `schema_version` only (no structural change).
- Version > 2: raises error (future-proofing).
- Version 1→2 migration: stub awaiting future work.

### 3.4 Template dispatch (build.py)

Template information lives in `design_tokens.yaml` under `templates:`:

```yaml
templates:
  cover:
    ref_index: 0
    capabilities: {has_body: false, has_blocks: false, body_clears: false}
    slots: {eyebrow: "Text 3", kicker: "Text 4", ...}
    logo: {left_in: 17.639, top_in: 0.177, width_in: 2.065, height_in: 1.02}
  content:
    ref_index: 1
    capabilities: {has_body: true, has_blocks: true, body_clears: true}
    slots: {title: "Text 1"}
    # ...
  closing:
    ref_index: 7
    capabilities: {has_body: false, has_blocks: false, body_clears: false}
    slots: {eyebrow: "Text 3", hero: "Text 4", ...}
```

Dispatch is keyed by template name — there is **no** `if tname == "content"`
branching anymore. Capability flags:
- `body_clears` = whether the body zone is stripped before composition.
- `has_blocks` = whether block rendering is allowed on this template.
- `has_body` = reserved for Phase C layout expansion.

### 3.5 Tokens system (`tokens.py`)

- `Tokens` class wraps the YAML dict with typed property accessors.
- `resolve_color(value)` resolves role names (`"primary"`) → hex (`"#1FB8B8"`).
  Passes through hex values starting with `#`.
- `brand_hexes()` returns the set of all 13 palette hexes (uppercased).
- `template(name)` returns the template entry from `design_tokens.yaml`.

### 3.6 Chrome replacement (`chrome.py`)

- **Minimum-overwrite principle**: only `run.text` is changed; the run's font,
  size, color, bold are inherited from the template clone. This is the mechanism
  that keeps every slide pixel-faithful.
- `apply_slots(slide, slots, fields)`: iterates `{field_key: shape_name}`
  mappings. Handles both scalar slots (`"title": "Text 1"`) and list slots
  (`"steps": ["Text 8", "Text 11", ...]`).
- Returns list of missing required field keys (slack for CLI error reporting).
- `set_slot_text()` strips extra runs/paragraphs from the first paragraph after
  replacing the text, keeping the clone clean.

### 3.7 Block rendering (`blocks.py`)

20 block kinds dispatched via `BUILDERS` dict:
`heading, body, bullets, caption, table, card, darkcard, steps, kpi, image, quote, separator, tags, badge, legend, timeline, flow, columns, feature_grid, comparison`

**Key design characteristics:**

- **All placement is free** (x, y, w in inches from deck.json).
- **All styling is system-bound**: every builder calls `style.py` helpers that
  force Montserrat, brand colors, type scale.
- **Body zone enforcement**: `_check_zone()` validates that every block is within
  `y_top_in = 1.2"` and `y_bottom_in = 10.5"`. Per-template overrides supported
  via a `_tname` parameter injected by `render_block()`.
- **Block-level defaults**: each builder sets sensible defaults (e.g. cards use
  `fill="white"`, accents use `color="primary"`).
- **Image block** (`add_image`): resolves src against cwd and `templates/media/`.
  Supports `fit=contain|cover|fill`. Uses Pillow (`PIL`) for aspect-ratio
  calculation — **Pillow is not declared in pyproject.toml**.
- **Flow connector lines**: raw lxml/XML construction for arrows between nodes
  (python-pptx gap).

### 3.8 Style system (`style.py`)

| Helper | Purpose |
|--------|---------|
| `hex_to_rgb()` | `"#1FB8B8"` → `RGBColor(0x1F, 0xB8, 0xB8)` |
| `style_run()` | Force font name, pt, bold, color on a single run |
| `style_text_frame()` | Style first paragraph/run + alignment + line spacing |
| `style_shape_solid_fill()` | Fill an autoshape with a brand hex |
| `no_line()` | Remove shape outline |
| `inches()` | Inches → EMU (914400 × inches) |

## 4. Current Unfinished/Stubbed Areas & Implementation Risks

### 4.1 Unfinished/stubbed

| Area | Location | Status |
|------|----------|--------|
| **`section_divider` template** | schema.py:87–88, design_tokens.yaml | **Always rejected** — the branded chrome reference slide does not exist in template.pptx. Phase D task 14. |
| **Layout expansion** | build.py:99–102 | Stubbed `LAYOUTS[layout_name].build(...)` — Phase C. Presently unused; all decks specify blocks directly. |
| **Schema migration v1→2** | schema.py:125 | Stub only. No structural v1→v2 differences exist yet, but the hook is ready. |
| **Page numbers** | design_tokens.yaml content template | `enabled: false` — resolved NOT to add, matching original template. |
| **SVG icon set** | style book mention | Referenced in style book prohibited list ("use the SVG icon set"), but no SVG handling exists in blocks.py. |
| **Font embedding** | ADR-0001, runbooks | PowerPoint UI only; no programmatic path. python-pptx cannot embed fonts (#355). |
| **Phase 2 (layout-fill)** | ADR-0001 Future | One-time PowerPoint authoring of named layouts; design_tokens.yaml schema stays unchanged. |

### 4.2 Implementation risks

| Risk | Severity | Details |
|------|----------|---------|
| **Missing dependency: Pillow** | **HIGH** | `from PIL import Image` in `blocks.py:add_image()` will fail at runtime if Pillow is not installed. Not listed in pyproject.toml dependencies. |
| **Template shape-name drift** | **MEDIUM** | Slot maps reference shape names (e.g. `"Text 1"`, `"Text 3"`). If the template.pptx is re-authored (even a rename in PowerPoint), slot replacement silently fails. Mitigated by `scripts/dump_tokens.py` + validator. |
| **px:NotesMaster missing** | **LOW** | `slide.notes_slide` requires a notes master. If the template lacks one, the archetype hint silently fails (wrapped in try/except). |
| **8-slide template assumption** | **LOW** | `n_orig = len(prs.slides._sldIdLst)` hardcodes the count. If template.pptx gains/loses slides, index shifts break clone references. |
| **Image path resolution** | **LOW** | `add_image()` tries three candidates (cwd, `templates/media/<src>`, `templates/media/<basename>`). If an engagement uses a path outside this scheme, images silently fail. |
| **Connector lines limited** | **LOW** | `add_flow` only supports straight connectors from right-center to left-center. No curved/elbow connectors. |
| **List slot length mismatch** | **LOW** | `apply_slots` with list slots (e.g. 5 cover steps) — if deck.json provides fewer items than template shape names, missing names are silently skipped. |
| **Validator EMU rounding** | **LOW** | Uses ±0.02" tolerance for position checks. At 914400 EMU/in this is ~18k EMU — unlikely to false-positive but edge-case coordinates near boundaries may pass. |

## 5. Recommendations: What Goes in README vs Deep Technical Report

### README (user-facing, skill-facing)

Keep current README content. Add:

1. **Pillow dependency** — document that Pillow is required for image blocks.
2. **Missing template error messages** — brief guidance on "section_divider not
   supported" and how to tell when the template has drifted.
3. **Block coordinate system** — a one-sentence reminder that x/y/w are inches,
   body zone is y=1.2→10.5, base margin is 0.6.

### Deep Technical Report (this document + docs/)

Keep in this artifact and in docs/:

1. **python-pptx limitations and workarounds** — the full four-item list from
   ADR-0001 plus connector XML workaround.
2. **Slot replacement minimum-overwrite principle** — why only `.text` is
   swapped (formatting is inherited from clone).
3. **Validator heuristic vs hint-based detection** — how `BAMI::template=...`
   hints in slide notes work and why they're authoritative.
4. **Capability flags** — the `body_clears`, `has_blocks`, `has_body` triple
   and how they drive generator branching.
5. **Risks table** — the full risk section above, for maintainer awareness.
6. **Template re-author workflow** — exact steps (`dump_tokens.py` → reconcile
   → regenerate → re-validate) for when the .pptx template changes.
7. **Phase B/Phase C gate criteria** — what would be required to enable
   `section_divider` (new reference slide in template) and `layout` expansion
   (a `LAYOUTS` registry).
