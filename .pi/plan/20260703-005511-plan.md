# Implementation Plan — Documentation for External Audit

## Goal
Produce two deliverables that together enable an independent external audit of presentation-framework: a deep technical/audit report (`docs/architecture/technical-description.md`) covering architecture, implementation details, dependencies, workflow, philosophy, template references, risks, constraints, and future direction; and an updated `README.md` that gives a shorter high-level conceptual overview while linking to the deep report.

## Tasks

### Deliverable 1 — Deep Technical/Audit Report

**1. Create directory structure for the deep report**
- File: `docs/architecture/` (new directory)
- Changes: create `docs/architecture/` alongside existing `docs/decisions/`, `docs/guidelines/`, `docs/runbooks/`
- Acceptance: `docs/architecture/` exists

**2. Write `docs/architecture/technical-description.md` — Part 1: Conceptual Architecture**
- File: `docs/architecture/technical-description.md` (new, ~600-800 lines)
- Content to include:
  - **Why this architecture exists**: python-pptx constraints (cannot create masters/layouts #413/#656, cannot embed fonts #355, cannot import layouts across files #1028), the decision to clone instead of create layouts
  - **Three-template model**: cover (hero opener, first slide only), content (workhorse, middle slides), closing (last slide only)
  - **Slide-clone mechanism**: deep-copy `<p:sld>` XML + image relationship remapping (`shared/pptx/clone.py`)
  - **Philosophical core**: "Composition may vary; the system does not." — explanation of why free composition + system-bound styling
  - **Data-flow diagram** (ASCII art): three inputs → build → branded.pptx → validate → deliver
  - **Pipeline stages**: parse inputs → load tokens → open template → resolve references → per-slide (clone → clear body zone → fill chrome slots → write notes hint → render blocks) → prune reference slides → save → validate
  - **File index table**: every source file with line counts and purpose

**3. Write `docs/architecture/technical-description.md` — Part 2: Template Assets & Token System**
- Content:
  - `templates/template.pptx` — 8-slide locked brand asset, structural facts (single layout, zero placeholders, Montserrat not embedded, stock Office theme colours, broken decorative icons)
  - `templates/design_tokens.yaml` — machine source of truth: canvas, 13-colour palette, type scale (20 sizes), grid (base 0.6"/fine 0.3"), per-template entries (ref_index, capabilities, logo positions, slot maps, footer definitions)
  - `tokens.py` — Tokens class, `resolve_color()` role→hex resolution, `brand_hexes()` for validator
  - Capability flags system: `body_clears`, `has_blocks`, `has_body` — how they drive generator dispatch (no `if tname == "content"` branching)
  - `templates/media/` — empty but reserved directory; image path resolution strategy
  - `templates/src/` — historical corpus (10 files, ~310 MB), 16:9 vs 4:3 legacy, Montserrat vs Montserrat Medium divergence

**4. Write `docs/architecture/technical-description.md` — Part 3: Runtime Pipeline Deep-Dive**
- Content:
  - **Clone engine** (`clone.py`): 4 steps (drop layout placeholders → deep-copy shapes → remap image rels → rewrite embed/link attributes), verified 53/53 shapes on template.pptx
  - **Slot replacement** (`chrome.py`): minimum-overwrite principle (only `run.text` changes, formatting inherited from clone), `apply_slots()` scalar vs list slot handling, `set_slot_text()` extra-run cleanup
  - **Body zone clearing** (`build.py._clear_body_zone()`): heuristic shape removal by `top` EMU bounds, fallback constants vs token-based zones, per-template override support
  - **Block rendering** (`blocks.py`): all 20 block kinds documented with parameters, defaults, behaviour, and which `style.py` helpers each uses
  - **Style system** (`style.py`): 6 helpers (`hex_to_rgb`, `style_run`, `style_text_frame`, `style_shape_solid_fill`, `no_line`, `inches`), how they force Montserrat/brand hex/type scale
  - **Schema system** (`schema.py`): JSON Schema loading, `_validate_semantics()` rules (first/last slide, cover/closing position, section_divider blocked, content title required), `_migrate_deck()` from legacy versions, `CURRENT_SCHEMA_VERSION = 2`
  - **Archetype hints**: `_write_archetype_hint()` writes `BAMI::template=<tname>` into slide notes; validator reads via `_read_archetype_hint()` for authoritative template detection, falls back to logo heuristics

**5. Write `docs/architecture/technical-description.md` — Part 4: Validator Internals**
- Content:
  - Full 11-check list: background, logo position, fonts (Montserrat only), colours (brand palette only), title bar (content slides), title text (content slides), footer (DELIVERING VALUE + Proprietary & Confidential), canvas bounds, round-trip, structure (first=cover, last=closing)
  - Archetype hint mechanism (authoritative) vs heuristic fallback (`_is_content()`/`_is_cover_like()` by logo EMU position)
  - Tolerance values: `GRID_TOL = 0.06"`, position `_near()` uses 0.02" tolerance, grid check intentionally not enforced (0.3" grid is guideline, not rule — template's own rhythm uses 8.6/18.8" bars)
  - Exit codes explained: 0 = pass, 1 = violations, plus CLI error codes from `pptx_gen` (file not found, schema error, build error)
  - How violations are reported: per-slide list with shape names, run text snippets, measured values

**6. Write `docs/architecture/technical-description.md` — Part 5: Dependencies, Risks, and Known Limitations**
- Content:
  - **Declared dependencies**: python-pptx==1.0.2, pyyaml≥6.0, jsonschema≥4.20, click≥8.1
  - **Critical missing dependency**: Pillow/PIL used by `add_image()` for aspect-ratio calculation (fit=contain/cover/fill) — NOT in pyproject.toml → runtime crash risk
  - **Risk table**:
    - HIGH: missing Pillow dependency
    - MEDIUM: template shape-name drift (slot maps reference `"Text 1"` by name)
    - MEDIUM: body-zone heuristic may strip future chrome inside the band
    - LOW: 8-slide template index assumption, notes master missing, connector line limitations, list-slot length mismatch, validator EMU rounding tolerance
  - **Known limitations**:
    - Montserrat NOT embedded (requires one-time PowerPoint UI step)
    - Two schema authorities (inline dict in schema.py + content-schema.json file) must be manually kept in sync
    - `section_divider` intentionally blocked — branded chrome slide not yet authored (Phase D task 14)
    - No SVG icon/media library (`templates/media/` empty)
    - No image relationship helpers (ad-hoc path resolution in `add_image()`)
    - Flow connector lines limited to straight connectors only (no curved/elbow)
    - Page numbers intentionally disabled (design decision, not omission)
    - Broken decorative icons in template slides 1–6 (cosmetic, not used by framework)

**7. Write `docs/architecture/technical-description.md` — Part 6: Extension Points, Migration, and Future Direction**
- Content:
  - **Adding a new block kind**: 4-step process (create builder in `blocks.py` → register in `BUILDERS` → add to schema `kind` enum → add to style book)
  - **Adding a new template**: requires designer editing template.pptx, `scripts/dump_tokens.py`, YAML entry with capabilities/slots/logo/ref_index, schema `TEMPLATE_NAMES` entry, semantic validation update
  - **section_divider escape hatch**: what would be required (new reference slide at index 8+, YAML registration, semantic validation relaxation, validator chrome assertions)
  - **Phase 2 layout-fill**: re-author three templates as named layouts with placeholders + branded theme + embedded Montserrat; `add_slide(layout)` + fill placeholders instead of clone; public schema unchanged → drop-in upgrade
  - **Template re-author workflow**: exact steps (replace template.pptx → `dump_tokens.py` → reconcile YAML → regenerate sample → re-validate)
  - **Migration notes**: 4:3→16:9 legacy conversion guidance, Montserrat Medium→Regular+Bold standardisation rationale, coordinate mapping for legacy content

**8. Write `docs/architecture/technical-description.md` — Part 7: Sample/Client Decks and Historical Corpus**

## Risks
- **Deep report length**: 600-800 lines of analytical prose is substantial. Ensure each section is self-contained with explicit cross-references so auditors can jump to relevant sections without reading the whole document
- **Dual schema authority not fixed**: Tasks 14-15 only add provenance comments. The actual fix (single authority) is a separate code refactor — the documentation must clearly flag this as a known maintenance hazard
- **Pillow dependency not declared**: documentation must warn about this clearly; it's a runtime crash waiting to happen for any user trying `image` blocks without Pillow installed
- **README length vs skimmability**: README will grow from ~50 lines to ~200-250 lines. The structure with clear section headers and the `##` hierarchy should preserve skimmability — external auditors can quickly find what they need
- **Artifact visibility**: `.pi/research/` artifacts remain outside the visible `docs/` tree. The plan adds cross-references in the deep report but does not move or publish them — this is intentional (`docs/` is checked in and public-facing; `.pi/` is agent-internal)
- **`section_divider` blocked status**: documentation must clearly state this is a temporary limitation pending manual template authoring (Phase D task 14), not a permanent architectural constraint
