# Implementation Plan — Envato Palette E2E Verification & Visual Audit

> Scope: verify the full BAMi presentation flow end-to-end, prove the `templates/media/reference/library`
> palette is (or is explicitly made) part of the runtime path, generate tests + a multi-slide showcase,
> run a full visual categorization audit, enforce customer isolation, and update docs/handoff.
> Every task is tagged **[VERIFY]** (read-only / test / artifact) or **[ARCH]** (changes runtime code) or **[GUARD]**.

## Goal

Establish a safe pre-change git snapshot, then produce evidence — tests, a showcase deck, an error log,
and a visual audit — that states precisely which Envato palette categories flow content→widget→render→validate today,
which are reference-only, and what must change for the palette to stop being a dead-end branch.

---

## Reality check (facts fixed before planning)

These were confirmed by reading the source and must be respected by every task:

- **Runtime block kinds (10):** `heading, body, bullets, caption, table, card, darkcard, steps, kpi, gantt`
  — `shared/pptx/blocks.py` `BUILDERS`, mirrored in `schemas/content-schema.json` `kind` enum and `shared/pptx/schema.py`.
- **Semantic layouts (3):** `gantt, comparison_panel, kpi_strip` — `shared/pptx/layouts.py` `LAYOUTS`.
- **LATENT DEFECT — `comparison_panel` is broken at render time:** `_layout_comparison_panel` emits a block
  with `kind: "comparison"`, but `"comparison"` is **not** in `BUILDERS`. `render_block` will raise
  `ValueError: unknown block kind 'comparison'`. This must be fixed or quarantined before any E2E claim.
- **`kpi_strip` silently drops fields:** layout forwards `delta`/`period` into the `kpi` block, but `add_kpi`
  ignores both — visual delta/trend rendering assumed by `test_blocks_new.py` does **not** exist.
- **Palette is a dead-end corpus today:** no runtime or validator code imports `templates/media/reference/library`.
  The only runtime "palette" is `templates/design_tokens.yaml` (brand tokens, not widgets).
- **`tests/test_blocks_new.py` is dead:** imports non-existent `_read_archetype_hint` from `tools.pptx_validate.cli`
  and parametrizes 11 phantom kinds (`image, quote, separator, tags, badge, legend, timeline, flow, columns,
  feature_grid, comparison`) absent from `BUILDERS`. It fails at collection — apparent per-kind coverage is not real.
- **Library = 93 PNGs / 18 non-empty categories / 2 empty** (`uncategorized`, and declared-but-missing `infographic-element`).
- **Build entrypoint:** `build_deck(deck_path, out_path, template_path, tokens_path) -> {slides_rendered,out,pruned}`.
- **Customer isolation today:** `.gitignore` blocks `clients/*` except `_sample/` + `clients/README.md`; no automated guard exists.

### Palette → runtime mapping (the central finding)

| Library category | Runtime path today | Status |
|---|---|---|
| `kpi` | `kpi` block / `kpi_strip` layout | ✅ live |
| `gantt` | `gantt` block / `gantt` layout | ✅ live |
| `card` | `card` / `darkcard` blocks | ✅ live |
| `table` | `table` block | ✅ live |
| `comparison` | `comparison_panel` layout | ⚠ BROKEN (emits unknown `comparison` kind) |
| `process` | none (nearest: `steps`) | ❌ reference-only |
| `timeline` | none | ❌ reference-only |
| `flow` | none | ❌ reference-only |
| `decision` | none | ❌ reference-only |
| `agenda` | none (nearest: `heading`+`bullets`) | ❌ reference-only |
| `project-status` | none (nearest: `kpi_strip`/`table`) | ❌ reference-only |
| `team`, `section-divider`, `use-case`, `background`, `executive-summary`, `project-charter`, `quote` | none | ❌ reference-only |
| `uncategorized`, `infographic-element` | n/a | empty / missing dir |

**Conclusion:** the user's expectation "all widget groups generate content" **cannot** be met by tests alone.
It is blocked on an architecture decision (see Decision Points D1–D2).

### Phase 1 — Rehabilitate the broken test surface (so E2E evidence is trustworthy)

3. **[ARCH] Quarantine or fix `tests/test_blocks_new.py`**
   - File: `tests/test_blocks_new.py`
   - Changes:
     - Split into two files: keep only the tests that exercise **real** kinds in `test_blocks_live.py`;
       move the dead tests into `tests/_disabled/test_blocks_phantom.py.disabled` with a header comment
       documenting that `_read_archetype_hint` and 11 phantom kinds do not exist in the runtime.
     - In the live half, restrict `_KINDS` to the 10 real kinds and drop the `delta`/`period` assertions
       (see Task 4) and the `comparison`/`image`/`quote` etc. cases.
   - Acceptance: `pytest tests/test_blocks_live.py` collects and passes; the disabled file is not collected.

4. **[ARCH] Decide the `comparison_panel` defect (BLOCKER)**
   - Files: `shared/pptx/layouts.py`, `shared/pptx/blocks.py`.
   - This is a required fix because the only layout that could back the `comparison` palette category is broken.
   - Two options (pick via Decision Point D3):
     - **(A) Minimal:** add an `add_comparison` builder that composes `card` panels (left/right), register in `BUILDERS`, add `"comparison"` to the schema `kind` enum.
     - **(B) Redirect:** change `_layout_comparison_panel` to emit `card` blocks directly (no new kind), keeping the layout name.
   - Acceptance: a `comparison_panel` layout slide builds and validates green; a parametrized test locks it.

5. **[ARCH] Fix `kpi` delta/period contract**
   - File: `shared/pptx/blocks.py`, `schemas/content-schema.json`, `shared/pptx/schema.py`.
   - Changes: either render `delta`/`period` (small caption + optional trend run) OR remove the fields from layout forwarding and document KPI as number+label only. Decision D4.
   - Acceptance: behavior matches schema; a test asserts the chosen contract.

### Phase 2 — E2E test strategy (prove the live flow end-to-end)

6. **[VERIFY] Add a per-live-kind parametrized build+validate test**
   - New file: `tests/test_palette_runtime_matrix.py`
   - For each kind in the 10 real kinds: build a 3-slide deck (cover + 1 content slide with that block + closing),
     `build_deck(...)`, then `validate(...)`; assert `rep.ok` and `slides_rendered == 3`.
   - Reuse the `_deck_with_blocks` helper pattern from the rehabilitated `test_blocks_live.py`.
   - Acceptance: 10 green parametrized cases; a new `comparison_panel`/`kpi_strip` case each.

7. **[VERIFY] Add a layout-dispatch test for all 3 layouts**
   - New file: `tests/test_layout_dispatch.py`
   - Cases: `gantt`, `comparison_panel` (post-fix), `kpi_strip` — assert build succeeds, validates, and
     deck opens round-trip.
   - Acceptance: 3 green cases.

8. **[VERIFY] Add negative-path tests**
   - New file: `tests/test_build_negative.py`
   - Cases: unknown block kind raises `ValueError` (covers the latent defect class); content slide missing
     `fields.title` raises; `gantt` with no `periods` raises; block crossing footer (`y+h > 10.5`) raises.
   - Acceptance: each raises the expected exception type.

9. **[VERIFY] Add CLI exit-code tests**
   - New file: `tests/test_cli_exit_codes.py`
   - Invoke `tools/pptx_validate/__main__.py` via `click.testing.CliRunner` on a valid deck (exit 0) and a
     mutated deck with a bad color (exit 1). Also test `tools/pptx_gen` build path exit codes.
   - Acceptance: exit codes asserted (currently untested per audit).

10. **[VERIFY] Add a schema-sync test that locks the kind enum to BUILDERS**
    - New file: `tests/test_kind_enum_sync.py`
    - Assert `set(schema.SCHEMA[...kind enum]) == set(BUILDERS.keys())` and equals the enum in

    - Acceptance: test fails loudly if the three sources diverge.

## Risks

- **R1 — Pytest collection flakiness:** audit reports "No tests collected" intermittently. Mitigate by running
  `pytest -p no:cacheprovider --co -q` to confirm collection before trusting results.
- **R2 — Showcase uses `_sample`**: a future agent could taint it with customer content. Mitigated by Task 17.
- **R3 — Mermaid/image paths:** not part of palette; keep showcase to vector blocks to avoid scope creep.
- **R4 — `deck_dir` traversal:** latent; only a doc note this pass (Task 18). Do not wire it to file reads.
- **R5 — Low-res references (16) and sparse categories (3)** may make the "all groups" showcase look thin;
  the reference-only stub deck (Task 13) documents this honestly rather than over-claiming.
- **R6 — The presentation skill itself** could pull defaults from engagement dirs; Task 17 guard + explicit
  `_sample` paths in the build invocation prevent contamination.

## Blockers (where expectation exceeds current architecture)

- **B1:** "All widget groups generate content" is impossible for 15/20 categories without new runtime widgets.
  No amount of testing closes this; it needs Decision D1.
- **B2:** "Palette participates in classification" (runtime widget lookup from the library) does not exist
  anywhere in code. This is a new subsystem, out of scope for a verification pass; needs Decision D2.
- **B3:** "Validation against the library reference" does not exist — the validator checks brand chrome only.
  Needs Decision D2 before it can be built.

## Decision Points (require user choice before proceeding past Phase 1)

- **D1 — Reference-only vs new runtime widgets:** For the 15 unmapped categories, choose:
  (i) keep them reference-only and document (fast, honest); or (ii) author "out-of-the-box" generic widgets
  (timeline, flow, process, decision, team, quote, agenda, section-divider, project-status, use-case,
  executive-summary, project-charter) — this is real ARCH work, multiple sprints.
- **D2 — Runtime palette bridge:** Build a `widget_lookup`/classification API so the generator queries
  `reference/library` (true palette participation) and the validator checks widget-vs-reference conformity.
  In/out of scope for this engagement? If in scope, it supersedes the "verification-only" framing.
- **D3 — `comparison_panel` fix style:** Option A (new `comparison` block kind) vs Option B (emit `card`s).
  Recommend **A** (matches library `comparison` category + the showcase intent).
- **D4 — `kpi` delta/period:** Render them (richer) or drop them (simpler). Recommend **render** to match the
  intended KPI widget, gated by Task 5.

> Default posture if decisions are not given: proceed **verification-only** — fix the two real defects
> (Tasks 4–5) so E2E evidence is honest, mark the 15 categories reference-only, and surface D1/D2 to the user
> in the handoff rather than silently expanding architecture.
