# Implementation Plan — BAMI Content Fabric

## Goal
Превратить репозиторий из генератора «только презентаций» в фундамент **BAMI Content Fabric** — брендового контент-фабрики, способной выпускать несколько типов клиентских документов (презентации, техдокументация, тендерные документы), — через безопасную поэтапную миграцию: централизация path-resolution → починка packaging → переименование репо в `bami-content-fabric` → мета-skill каркас, без преждевременной глобализации runtime и без большого рефакторинга структуры.

> **Этот план заменяет** `20260703-201726-global-skill-plan.md`. Прежнее направление («вынести `presentation-design` SKILL.md в глобальный каталог с cwd-guard») **отменено** новыми research-данными: доменные skills должны оставаться локальными в репозитории (жёсткая связь с путями/шаблонами/CLI). См. раздел «Глобализовать ли `presentation-design` сейчас».

## Key recommendation (читать первым)

**Немедленный repo-wide рефакторинг структуры (`shared/fabric/`, `shared/docx/`, перенос блоков в домены) — НЕ делать.** Это рискованно и преждевременно: второй домен (техдокументация) ещё не нужен реальному engagement'у, а абстракции без второй реализации = architecture astronautism (см. research `bami-fabric-structure` §5.7).

**Preferred approach (staged, now):** «Локальные домены + централизация путей + отложенный rename» —
1. Сначала убираем хрупкие `parents[2]`-хаки через единый `_paths.py` + маркер `.bami-root` (делает репо переименовываемым).
2. Чиним `pyproject.toml` до cleanly-installable.
3. Ставим локальный мета-skill `content-fabric` с `registry.yaml` (каркас под будущие домены) и добавляем cwd-guard + path-зависимости в существующий `presentation-design` skill (он остаётся **локальным**).
4. Переименовываем репо `presentation-framework` → `bami-content-fabric` + обновляем ~22 identity/metadata-файла (теперь безопасно благодаря централизованным путям).
5. Fabric-core extraction и второй домен — **только когда появится реальный engagement** (Stage 4+, вне этого плана).

**Backup approach (быстрее к новой identity, но оставляет хрупкость дольше):** переименовать директорию **сейчас** (это безопасно, т.к. глубина вложенности не меняется и `parents[2]` продолжает резолвиться), обновить metadata-файлы, а централизацию path-resolution и packaging отложить до fabric-core extraction. Плюс: быстрее получаем имя `bami-content-fabric`. Минус: `parents[2]`-хаки остаются единственным якорем и ломаются при любом будущем переносе runtime.

Подробнее — ниже в «Preferred vs Backup».

## Should we globalize `presentation-design` now? — НЕТ

**Решение: НЕ глобализовать `presentation-design` (ни SKILL.md, ни runtime) сейчас.** Это прямая отмена прежнего плана.

Обоснование (research `bami-fabric-skills` §2.2, §3.2):
- **CWD-binding.** Команды skill (`python -m tools.pptx_gen ...`) требуют `cwd = repo-root`; глобальный skill теряет эту гарантию и вынуждает хардкодить абсолютные пути под конкретную машину (непортативно между workstation/CI/контейнерами).
- **Tight coupling to repo artifacts.** Skill ссылается на `templates/template.pptx`, `schemas/content-schema.json`, `clients/`, `docs/` по относительным путям + требует проектный `node_modules/.bin/mmdc` + Chromium. Глобальная копия = либо хардкод абсолютных путей, либо runtime cwd-guard — оба варианта хрупки.
- **Доменные skills — repo-scoped по природе.** `technical-document-design`, `tender-document-design` будут ссылаться на тот же репо (`shared/docx/`, `templates/documentation/`, `schemas/`). Репозиторий И ЕСТЬ content fabric; доменные skills живут в его `.pi/skills/`.

**Что глобализовать можно (опционально, будущий cross-cutting skill):** `bami-design-system` — чисто знания (палитра, шрифты, tone-of-voice), применимые ко ВСЕМ репо BAMi, без файловых зависимостей от этого репо. Это единственный кандидат на глобальный навык. **В рамках этого плана его не создаём** — отдельное решение.

**Что нужно сделать с `presentation-design` сейчас (вместо глобализации):** оставить локальным, добавить **cwd-guard** и секцию **path-зависимостей** (Stage 2). Это делает его безопасным внутри репо и готовым к будущему meta-skill dispatch'у.

## Repo naming / repositioning strategy

**Целевое имя:** `bami-content-fabric` (research §1: «fabric» = переплетение нескольких выходов из общих нитей; не пере-обещает конкретный тип документа, в отличие от `presentation-framework`/`bami-content-factory`).

**Порядок критичен** (research `bami-fabric-rename-impact`):
1. Централизация path-resolution (Stage 0) — **prerequisite** для безопасного любого переноса.
2. Packaging-фикс (Stage 1) — prerequisite для будущей дистрибуции.
3. Rename + identity-обновление (Stage 3) — теперь безопасно.

**Переименование директории само по себе безопасно** (глубина файлов не меняется → `parents[2]` продолжает работать) — но только до тех пор, пока runtime не переносится в другую иерархию (например в `~/.pi/agent/skills/`). Поэтому переименование делаем, а перенос runtime — нет.

**Почему `bami-content-fabric` а не `bami-content-factory`:** fabric подразумевает взаимосвязанные, композируемые capabilities (skills + templates + pipelines + media); factory — одностороннее производство. Имя репо становится зонтичным идентификатором; `presentation-framework` остаётся только в deprecated-ссылках/истории.

## Skill portfolio strategy

| Skill | Тип | Расположение | Статус |
|---|---|---|---|
| `presentation-design` | Доменный | **локально** `<repo>/.pi/skills/` | существует; Stage 2 = guard + path-docs + frontmatter `fabric:` |
| `content-fabric` | **Мета/routing** | **локально** `<repo>/.pi/skills/` | NEW (Stage 2); каркас + `registry.yaml` |
| `technical-document-design` | Доменный | локально (будущее) | Stage 4 — когда нужен реальный engagement |
| `tender-document-design` | Доменный | локально (будущее) | Stage 5 — по мере необходимости |
| `bami-design-system` | Cross-cutting | глобально `~/.pi/agent/skills/` (будущее) | опционально, вне этого плана |

**Naming convention:** `<document-family>-design` для доменов; `content-fabric` (без суффикса) для мета-skill. Антипаттерны: `doc-design` (двусмысленно), `presentation-skill` (непоследовательно), `bami-presentation` (избыточный префикс).

**Collision avoidance:** мета-skill `content-fabric` — единая точка входа с широким description; читает `registry.yaml`, матчит пользовательский интент по `detects[]` и делегирует доменному skill. Доменные skills имеют узкие description, чтобы не перехватывать чужой интент. Убираем потребность в прямом global-local дублировании (и коллизии first-wins из прежнего плана).

## Modular structure direction (только НАПРАВЛЕНИЕ, не сейчас)

Целевая архитектура (research `bami-fabric-structure` §2): document-agnostic core `shared/fabric/` (schema loader, tokens, template inheritance, slots, pipeline, mermaid, media, boilerplate, validate, output) + доменные модули `shared/pptx/`, `shared/docx/`, `shared/tender/` + shared design system `shared/design/core_tokens.yaml`. **Но это Stage 4+ — не выполнять сейчас.** Ближайшая задача — только проложить рельсы (централизация путей, packaging, meta-skill каркас, rename), чтобы будущий fabric-core extraction был инкрементальным и низкорисковым. Доказывать каждый домен реальным engagement'ом (anti-pattern §5.5: не создавать домен до реальной потребности).

## Preferred vs Backup

**Preferred — «Centralize-first staged migration»** (рекомендую): Stage 0 → 1 → 2 → 3 в указанном порядке. Каждый этап независимо shippable, низкий риск, rename безопасен благодаря централизованным путям. Минус: больше шагов до новой identity.

**Backup — «Rename-first»**: переименовать директорию сразу (безопасно при неизменной глубине) + обновить metadata, централизацию путей/packaging/meta-skill — позже. Плюс: быстрее к имени `bami-content-fabric`. Минус: `parents[2]` остаётся хрупким якорем; любой будущий перенос runtime сломается; оставляет техдолг. Выбирать, если срочность ребрендинга выше приоритета стабильности.

## What to change NOW vs what NOT to change yet

**Делаем сейчас (Stages 0–3):** централизация path-resolution; packaging-фикс; мета-skill `content-fabric` каркас + registry; cwd-guard/path-docs в `presentation-design`; rename репо + identity-файлы.

**НЕ делаем пока (явно):**
- ❌ Fabric-core extraction (`shared/fabric/`, перенос `tokens.py`/`mermaid_render.py`/`schema.py` в core) — Stage 4.
- ❌ Перестройка доменов (`shared/pptx/` → вынос blocks/layouts в доменный пакет; `tools/pptx_gen/` → `tools/pptx/gen.py`) — Stage 3 из research, но отложить до fabric extraction, чтобы не двигать файлы дважды.
- ❌ Разделение `design_tokens.yaml` на `core_tokens.yaml` + `tokens_ext.yaml` — Stage 2 research, отложить до второго домена.
- ❌ Домены `docx`/`tender` — Stage 4–5, только под реальный engagement.
- ❌ Глобализация `presentation-design` (runtime или SKILL.md) — отменено (см. выше).
- ❌ Удаление `sys.path`-хака из CLI как жёсткое требование — оставить обратно-совместимым; централизуем, но не ломаем.

## Rollback strategy
**Stages 0–2 (до rename) — изолированный откат:**
1. Runtime меняется только в Stage 0 (`_paths.py` + замены `parents[2]`). Откат: `git revert <paths-commit>` или `git checkout <pre-stage0-SHA> -- shared/ tools/`.
2. Stage 1/2 — отдельные коммиты; `git revert` по каждому.

**Stage 3 (rename) — откат rename:**
1. Вернуть имя директории `bami-content-fabric` → `presentation-framework` (или `git checkout <pre-rename-SHA>` для содержимого + ручной rename папки).
2. Identity-коммиты откатываются `git revert <rename-subcommits>`.
3. Так как Stage 0 централизовал пути, `project_root()` работает при любом имени папки → переименование обратно не ломает runtime.

**Глобальный rollback-якорь:** SHA из Task 8 (`git rev-parse HEAD` перед rename). Runtime-ассеты (`template.pptx`, `design_tokens.yaml`, `content-schema.json`, media) в Stage 0–3 **не удаляются и не переносятся** — риск потери данных минимален.
