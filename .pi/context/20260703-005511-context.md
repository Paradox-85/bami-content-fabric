### Code Workflow
Source artifact: `.pi/research/20260703-005511-code-workflow.md`

Key findings:
- End-to-end runtime pipeline:
  1. `deck.json` loaded and validated by `shared/pptx/schema.py`
  2. tokens loaded from `templates/design_tokens.yaml`
  3. template deck opened via `python-pptx`
  4. source slides resolved by `ref_index`
  5. per slide: clone → clear body zone → apply chrome slots → write notes hint → render blocks
  6. original reference slides pruned
  7. output saved
  8. validator re-opens the generated `.pptx` and enforces brand rules
- Important implementation mechanisms that docs should explain:
  - **minimum-overwrite slot replacement** in `chrome.py`: only `run.text` changes; formatting is inherited from the cloned run
  - token system in `tokens.py`
  - capability-based template dispatch in `build.py`
  - body-zone checks in block builders
  - validator reading archetype hints from slide notes before heuristic fallback
- Dependencies discovered:
  - declared: `python-pptx`, `pyyaml`, `jsonschema`, `click`, dev: `pytest`, `ruff`
  - important risk: Pillow (`PIL`) is used by `add_image()` and should be documented because image functionality depends on it
- Current unfinished / audit-relevant areas:
  - `section_divider` intentionally blocked pending manual template work
  - `layout` / `variant` / `content` semantic expansion remains scaffolded rather than fully implemented
  - there is no SVG icon/media library yet (`templates/media/` is empty)
  - migration path to future layout-fill architecture remains documented but not implemented
- Recommendation split:
  - README should mention core pipeline, dependency caveats, coordinate system, validator gate
  - deep technical report should hold internal mechanics, risks, extension points, and validator internals
